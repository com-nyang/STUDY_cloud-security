# Shared Responsibility Model

## One-Line Definition

클라우드 보안은 CSP(Cloud Service Provider)와 고객이 각자 다른 범위의 책임을 나눠 가지는 구조다.

## Why It Matters

이 모델을 이해하지 못하면 "클라우드 사업자가 다 막아주겠지"라는 잘못된 가정을 하게 된다. 실제로 많은 사고는 클라우드 자체 취약점보다 고객의 잘못된 설정, 과도한 권한, 공개 스토리지, 부실한 모니터링에서 발생한다.

## Core Structure

- CSP 책임: 물리 인프라, 하드웨어, 글로벌 네트워크, 하이퍼바이저 등
- 고객 책임: 계정, 권한, 데이터, 네트워크 설정, 운영체제 설정, 애플리케이션 설정 등

서비스 모델에 따라 고객 책임 범위가 달라진다.

- IaaS: 고객 책임이 가장 많다.
- PaaS: 운영체제 관리 일부를 CSP가 맡는다.
- SaaS: 애플리케이션 자체 운영은 CSP 비중이 크지만, 계정/접근통제/데이터 사용 책임은 여전히 고객에게 남는다.

## Security Questions To Ask

공부할 때 아래 질문으로 정리한다.

1. 이 자산은 누가 관리하는가?
2. 이 설정을 내가 직접 해야 하는가?
3. 잘못 설정하면 어떤 보안 사고가 생기는가?
4. 사고가 나면 어떤 로그로 확인할 수 있는가?

## Common Misunderstandings

- "클라우드에 올리면 자동으로 안전하다"
- "관리형 서비스는 보안 설정도 자동이다"
- "사업자가 책임지니 공개 설정 사고도 사업자 책임이다"

이 세 가지는 전형적인 오해다. 실제로 계정 탈취, 과한 권한, 공개 버킷, 잘못된 네트워크 노출은 대부분 고객 책임 영역이다.

## Examples

- S3 버킷이 공개되어 민감 데이터가 노출됨: 고객 설정 책임
- MFA 없이 루트 계정을 운영함: 고객 계정 보호 책임
- 보안 로그를 켜지 않아 사고 추적이 안 됨: 고객 운영 책임
- 데이터센터 물리 접근 통제: CSP 책임

## Operational Takeaways

- 새 서비스를 볼 때 기능보다 먼저 책임 경계를 본다.
- "누가 패치하나?"와 "누가 접근통제 설정하나?"를 먼저 확인한다.
- 관리형 서비스라도 IAM, 네트워크, 데이터 보호 설정은 직접 검토한다.

## Detection Angle

Shared Responsibility Model 자체는 로그 항목이 아니라 운영 원칙이다. 다만 실제 탐지는 아래 영역과 연결된다.

- IAM 변경 로그
- 감사 로그 활성화 여부
- 공개 리소스 탐지
- 보안 기준선 위반 탐지
